# First review flow

Use when the Review Plan has `route=first_review`.

1. Explain in one or two sentences that the engine computed the numbers locally and that the review needs motive confirmation before it can produce a conclusion.
2. Ask every required question in `question_queue` in order, once, following the native-control, fixed-text, and presentation-trace protocols in `references/interaction-delivery.md`. A first review carries three to five highest-information items and may include an `initial_thesis` capture that asks, for a currently un-thesised holding, whether the entry followed a planned thesis, chased momentum, came from an external recommendation, or had no clear thesis. For `add_thesis`, `initial_thesis`, `headline_motive`, and `exit_consistency`, bind a grounded private surface through `review.py resume --question-surfaces`, then present the returned exact presentation; on any surface failure, keep the unchanged engine fallback. An `exit_consistency` question is the answerable form of the Block-3 `[?]` exit-opportunity-cost observation, asking whether the early exits it names were a deliberate rule, an emotional reaction, or an external constraint; the read-only panel yields to it when queued. A recent-exit question stays engine-rendered, captures a reason without judging the outcome, and saves `skip` so it is not asked again. Never merge, add, skip, or reorder questions; change canonical options or payload gates; invent facts; or use more than the one frozen clarification. A quoted thesis keeps its engine-declared voice. One fact-grounded observation per review may still enter `observations` without changing the queue.
3. Create an inferred thesis for every entry in `missing_thesis_positions`, following the field vocabulary in `review_plan.authoring_contract`. Submit the unchanged `cycle_id` plus the qualitative fields:
   - `why`: the fact or expectation that may not be priced in, or an honest placeholder such as "averaging down while waiting to recover; confirmation needed"
   - `exit_trigger`: a factual condition that would falsify the thesis, not a stop-loss price
   - `horizon`: a stable id from `card_plan.horizon_ids`, or null when no reasonable inference is possible (legacy Chinese values in old records must not be newly authored)
   The engine prefills the mechanical fields it owns (`ticker`, `maturity:"inferred"`, and snapshot provenance); never present an inferred thesis as user-confirmed. Optional fields (`stop`, `target_size`, `driver`, and the inference-only accumulation fields listed in `authoring_contract`) follow one rule: fill one only when the conversation contains a real signal, leave it null otherwise, and never ask extra questions for it â€” these cannot be backfilled later. Upgrade `_inferred` to false or `source_confidence` to `"confirmed"` only when the user volunteers it in an existing answer.
4. Keep the narrative qualitative. Write `headline` and `mirror`; optionally add `counterfactual` and `strength`. Do not write `rule_rationale`: the card no longer renders it (#301 â€” the engine derives the rule's trade-off sentence itself), and the field is retained only so older sessions stay readable. Cover every `card_plan.required_honesty_keys` entry with one sentence in `narrative.honesty` (wording guidance in `card-spec.md`). Frame coverage gaps the engine never asked about as neutral facts per `authoring_contract.narrative.unprompted_gaps`, never as the central judgment of the headline or mirror. Do not include digits.
5. Run preview. If validation fails, fix the artifact described by the error; do not bypass the gate.
6. Show the review-card preview inline following `references/card-delivery.md`, record the actual presentation following `references/interaction-delivery.md`, and only then ask the user to choose one candidate rule, provide a custom rule, or skip. Present each candidate's engine-authored `grounding` sentence verbatim with its rule when the payload carries one, so the choice references the user's actual positions; never invent a grounding for a candidate that has none. Recording `rule_choice_presented` requires the grounding-check evidence described in `references/interaction-delivery.md` (#293); `verify` fails closed if it is missing or shows a non-verbatim match. If the user states their own single-position cap here (for example "my limit is 25%"), record it with `review.py set-cap` per `references/data-contract.md` so future reviews reconcile against their number; do not add a card note explaining the threshold.
7. Write the choice to `answers.commitment`, then finalize. Return the review card (`card-private.md`) per the same delivery contract. Return the share-safe artifact only when the user asks for it; do not imply that the product publishes it.

Success means that a canonical session is committed and the user sees one card. Projection errors are repairable and must not be described as session loss.
